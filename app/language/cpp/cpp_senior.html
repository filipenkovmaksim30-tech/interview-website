Вопросы для C++ Senior Backend разработчика

  1. Что такое SFINAE и как оно используется в современном C++? Приведите примеры из стандартной библиотеки.

    SFINAE (Substitution Failure Is Not An Error) — принцип, при котором ошибка подстановки шаблонного параметра не приводит к ошибке компиляции, а лишь исключает данную перегрузку из рассмотрения. Используется для условной компиляции на основе свойств типов. В современном C++ часто заменяется на if constexpr и концепты (concepts из C++20), но классическое SFINAE лежит в основе std::enable_if, которое применяется в стандартной библиотеке для выбора перегрузок (например, разные реализации std::copy для тривиально копируемых типов и сложных объектов). Позволяет писать гибкий обобщённый код, адаптирующийся под свойства типов на этапе компиляции.

  2. Какие проблемы возникают при проектировании высоконагруженных многопоточных систем на C++? Расскажите про ложное разделение кэша (false sharing), ABA-проблему и способы их решения.

    False sharing — ситуация, когда разные потоки изменяют разные переменные, попавшие в одну кэш-линию процессора (обычно 64 байта), вызывая постоянную инвалидацию кэша и резкое падение производительности. Решение: выравнивание данных (alignas), добавление padding между независимыми атомарными переменными. ABA-проблема возникает в lock-free структурах: поток читает значение A, другой меняет A→B→A, первый поток по CAS ошибочно считает, что ничего не изменилось. Решение: tagged pointers (совмещение указателя со счётчиком версий), double-word CAS. Senior разработчик должен понимать эти тонкости для создания действительно эффективных конкурентных структур.

  3. Как организовать управление памятью в крупном C++ проекте? Сравните подходы: сырые указатели, умные указатели, кастомные аллокаторы, пулы памяти.

    Выбор стратегии зависит от характера проекта. Сырые указатели допустимы только для невладеющих ссылок (observer pattern). std::unique_ptr — для единоличного владения (90% случаев). std::shared_ptr — когда владение действительно разделяемое, но осторожно: циклические ссылки создают утечки (решение — std::weak_ptr). Кастомные аллокаторы реализуют специфичные стратегии выделения: пулы памяти (объекты одного размера), арены (выделение пачками с массовым освобождением), стековые аллокаторы. В высоконагруженном backend пулы и арены критичны: снижают фрагментацию, улучшают локальность данных (cache locality) и уменьшают накладные расходы на malloc/free. Важно профилировать и измерять эффект от внедрения аллокатора.

  4. Что такое CRTP (Curiously Recurring Template Pattern)? Приведите пример использования и объясните преимущества перед виртуальными функциями.

    CRTP — паттерн, при котором класс наследуется от шаблонного класса, параметризованного самим этим классом: class Derived : public Base. Позволяет реализовать статический полиморфизм: вызов методов базового класса разрешается на этапе компиляции, без накладных расходов на vtable и косвенные вызовы. Пример: std::enable_shared_from_this использует CRTP для корректного возврата shared_ptr на себя. Другой пример — mixin-классы для добавления функциональности (операторы сравнения, счётчики экземпляров). Преимущества: нулевые накладные расходы на вызов, возможность инлайнинга, более строгая проверка типов. Недостаток: нельзя хранить разнородные объекты в одном контейнере, как с виртуальными функциями.

  5. Какие нововведения C++17 и C++20 наиболее полезны для backend-разработки? Приведите конкретные примеры применения.

    C++17: std::optional — для функций, которые могут не вернуть значение (поиск в кэше, валидация), заменяет специальные значения-маркеры. std::variant — типобезопасная альтернатива union для обработки разных типов ответов (JSON может быть строкой, числом, массивом). Structured bindings — удобное разложение пар/кортежей (итерация по map). if constexpr — условная компиляция без SFINAE в шаблонах. std::string_view — избегает копирования строк при парсинге HTTP-заголовков. C++20: корутины (co_await, co_yield) — асинхронный код без callback hell, особенно полезны для сетевых операций (ожидание ответа от БД, HTTP-клиенты). Концепты (concepts) — ограничения шаблонных параметров, улучшают читаемость и сообщения об ошибках. std::span — легковесное представление непрерывного диапазона, удобен для работы с буферами данных. Диапазоны (ranges) — конвейерная обработка данных в функциональном стиле, код чище и понятнее.
