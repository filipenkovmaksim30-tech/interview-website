Вопросы для Java Senior Backend разработчика

  1. Как работает сборщик мусора (Garbage Collector) в JVM? Сравните разные типы сборщиков (Serial, Parallel, CMS, G1).

    Сборщик мусора автоматически освобождает память, занятую недостижимыми объектами. Serial — однопоточный, подходит для простых приложений. Parallel — многопоточный, ориентирован на пропускную способность. CMS (Concurrent Mark Sweep) — минимизирует паузы, работает конкурентно, но подвержен фрагментации. G1 (Garbage First) — делит heap на регионы, балансирует между пропускной способностью и паузами, сжимает память, рекомендован для больших куч (более 4 ГБ). Выбор сборщика зависит от требований к задержкам и объёму памяти.

  2. Как реализовать надёжную распределённую транзакцию в микросервисной архитектуре? Объясните паттерн Saga.

    В микросервисах ACID-транзакции через несколько сервисов невозможны из-за отсутствия общей базы. Saga — паттерн распределённой транзакции, состоящей из последовательности локальных транзакций с компенсирующими действиями. Типы: оркестрация (центральный координатор вызывает сервисы) и хореография (сервисы обмениваются событиями). При сбое выполняются компенсационные транзакции для отката выполненных шагов. Обеспечивает eventual consistency.

  3. Расскажите о механизме ClassLoader в Java. Какие типы загрузчиков существуют и в чём их взаимодействие?

    ClassLoader загружает байт-код классов в JVM. Иерархия: Bootstrap ClassLoader (базовые классы rt.jar), Platform ClassLoader (расширения Java SE), Application ClassLoader (классы из classpath). Работает по принципу делегирования: сначала запрос передаётся родителю, если класс не найден, текущий загрузчик пытается загрузить сам. Пользовательские ClassLoader’ы позволяют загружать классы из нестандартных источников (сеть, шифрованные jar), реализуют горячую замену.

  4. Какие паттерны проектирования вы используете в Spring-приложениях? Приведите примеры реализации.

    Spring широко использует: • IoC/DI (Inversion of Control / Dependency Injection) — управление зависимостями через контейнер. • Singleton — бины по умолчанию являются синглтонами в контексте Spring. • Proxy — для АОП (транзакции, логирование). • Template Method (JdbcTemplate, RestTemplate). • Factory (BeanFactory). • Observer (ApplicationEvent). Пример: при объявлении бина с аннотацией @Service Spring создаёт прокси для внедрения транзакций через @Transactional — это Proxy + Interceptor.

  5. Что такое реактивное программирование и как оно реализовано в Spring WebFlux? Когда его следует применять?

    Реактивное программирование основано на асинхронных неблокирующих потоках данных (Project Reactor, Mono/Flux). Spring WebFlux — реактивный web-фреймворк, работает на сервере Netty или контейнерах Servlet 3.1+. Преимущества: высокая пропускная способность при большом числе одновременных соединений, низкое потребление потоков. Подходит для I/O-bound операций (микросервисы, API шлюзы), но не даёт выигрыша для CPU-bound задач. Требует перехода всей цепочки доступа к данным на реактивные драйверы (R2DBC, Reactive MongoDB).
