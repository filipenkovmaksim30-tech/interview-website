Вопросы для Go Middle Backend разработчика

  1. Как работает сборщик мусора (GC) в Go? Какие подходы снижают его влияние на производительность?

    Начиная с Go 1.8 используется конкурентный сборщик мусора с пометкой и очисткой (mark-and-sweep), основанный на трёхцветном алгоритме. Он работает параллельно с приложением, минимизируя паузы (Stop-The-World только на короткую фазу). Для снижения нагрузки рекомендуется: переиспользовать объекты (sync.Pool), выделять память в стеке (избегать побега указателей в кучу), настраивать GOGC (процент роста кучи до запуска GC) в высоконагруженных системах.

  2. Что такое контекст (context.Context) и как его правильно использовать в backend-сервисах?

    Контекст предназначен для передачи дедлайнов, сигналов отмены и значений, связанных с запросом, через цепочку вызовов. Правила: контекст всегда передаётся первым аргументом функции; никогда не храните контекст в структуре (он временный); используйте context.WithTimeout/WithDeadline для HTTP-обработчиков, чтобы не держать зависшие соединения; context.WithValue — только для служебных данных (трассировка, логирование), не для бизнес-параметров.

  3. Опишите основные паттерны конкурентности в Go: fan-in, fan-out, pipeline.

    Fan-out — одна горутина отправляет задачи нескольким рабочим горутинам через каналы (например, распараллеливание обработки). Fan-in — несколько горутин сливают результаты в один канал (агрегация). Pipeline — последовательность этапов обработки, где каждый этап — горутина, получающая данные из входного канала и отправляющая результат в выходной. Эти паттерны позволяют строить эффективные конвейеры обработки данных, характерные для высоконагруженных серверов.

  4. Как устроен HTTP-сервер на стандартной библиотеке net/http? Как происходит обработка запроса?

    При запуске http.ListenAndServe создаётся TCP-листенер. Для каждого входящего соединения запускается отдельная горутина. Запрос парсится в http.Request, и маршрутизатор (http.ServeMux или любой другой, реализующий http.Handler) выбирает обработчик. Обработчик получает ResponseWriter для формирования ответа. Всё это работает конкурентно, поэтому необходимо защищать разделяемые данные (мьютексы, каналы).

  5. Расскажите про обработку ошибок в Go: обёртывание (wrapping), errors.Is, errors.As, fmt.Errorf и лучшие практики.

    В Go ошибки — это значения. С Go 1.13 введено обёртывание: fmt.Errorf("... %w", err) создаёт цепочку ошибок. errors.Is проверяет, есть ли в цепочке конкретная ошибка (сентинельное значение). errors.As извлекает ошибку определённого типа из цепочки. Лучшие практики: не игнорируйте ошибки; добавляйте контекст при возврате ошибки вверх; не оборачивайте ошибку без надобности; проверяйте ошибки на уровне, где можно принять решение (retry, fallback).
